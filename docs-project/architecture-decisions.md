# Архитектурные решения - Chewback

## Введение

Этот документ описывает ключевые архитектурные решения, принятые в проекте Chewback. Каждое решение включает контекст, рассмотренные варианты, выбранное решение и последствия.

## ADR-001: Выбор Tauri вместо Electron

**Дата:** 2024-01-15  
**Статус:** Принято  
**Участники:** Команда разработки

### Контекст

Требовалось выбрать фреймворк для десктопного приложения мессенджера с учетом:
- Безопасности хранения данных
- Производительности
- Размера приложения
- Кроссплатформенности

### Рассмотренные варианты

1. **Electron:**
   - Плюсы: Большое сообщество, много готовых компонентов
   - Минусы: Большой размер (~100MB), высокая потребность в памяти

2. **Tauri 2.0:**
   - Плюсы: Малый размер (~5MB), нативная производительность, безопасность
   - Минусы: Меньше готовых компонентов, более новая технология

3. **Flutter Desktop:**
   - Плюсы: Единая кодовая база для всех платформ
   - Минусы: Менее зрелый для десктопа, больший размер

### Решение

Выбрать **Tauri 2.0** по следующим причинам:
1. **Безопасность:** Встроенная система разрешений, изоляция кода
2. **Производительность:** Нативный Rust бэкенд, меньший overhead
3. **Размер:** Значительно меньший размер дистрибутива
4. **Совместимость:** Поддержка всех основных платформ

### Последствия

**Положительные:**
- Меньший размер приложения
- Лучшая производительность
- Улучшенная безопасность
- Современная архитектура

**Отрицательные:**
- Необходимость изучения Rust
- Меньше готовых UI компонентов
- Более сложная отладка нативных компонентов

## ADR-002: Использование SolidJS вместо React

**Дата:** 2024-01-20  
**Статус:** Принято  
**Участники:** Команда разработки

### Контекст

Требовалось выбрать фронтенд фреймворк для Tauri приложения с учетом:
- Производительности
- Размера бандла
- Простоты интеграции с Tauri
- TypeScript поддержки

### Рассмотренные варианты

1. **React:**
   - Плюсы: Большое сообщество, много библиотек
   - Минусы: Virtual DOM overhead, больший размер

2. **SolidJS:**
   - Плюсы: Отличная производительность, малый размер, реактивность без Virtual DOM
   - Минусы: Меньшее сообщество

3. **Vue 3:**
   - Плюсы: Composition API, хорошая производительность
   - Минусы: Больший размер чем SolidJS

### Решение

Выбрать **SolidJS** по следующим причинам:
1. **Производительность:** Отсутствие Virtual DOM, тонкая реактивность
2. **Размер:** ~7KB против ~40KB у React
3. **Интеграция:** Простая интеграция с Tauri командами
4. **TypeScript:** Отличная поддержка TypeScript

### Последствия

**Положительные:**
- Высокая производительность UI
- Малый размер бандла
- Простая реактивность
- Хорошая TypeScript интеграция

**Отрицательные:**
- Меньше готовых компонентов
- Необходимость изучения новой парадигмы
- Меньше ресурсов для обучения

## ADR-003: Rust бэкенд с Axum

**Дата:** 2024-01-25  
**Статус:** Принято  
**Участники:** Команда разработки

### Контекст

Требовалось выбрать технологию для серверной части с учетом:
- Безопасности
- Производительности
- Масштабируемости
- Поддержки WebSocket

### Рассмотренные варианты

1. **Node.js + Express:**
   - Плюсы: Большое сообщество, простота разработки
   - Минусы: Меньшая производительность, проблемы с памятью

2. **Go + Gin:**
   - Плюсы: Хорошая производительность, простота
   - Минусы: Меньшая типобезопасность

3. **Rust + Axum:**
   - Плюсы: Максимальная производительность и безопасность
   - Минусы: Более крутая кривая обучения

### Решение

Выбрать **Rust + Axum** по следующим причинам:
1. **Безопасность:** Гарантии безопасности памяти
2. **Производительность:** Нативный код, async/await
3. **WebSocket:** Отличная поддержка через tokio-tungstenite
4. **Типобезопасность:** Полная типобезопасность от БД до API

### Последствия

**Положительные:**
- Максимальная производительность
- Гарантии безопасности
- Отличная поддержка async
- Будущее-proof архитектура

**Отрицательные:**
- Сложность найма разработчиков
- Более долгая разработка
- Сложность отладки

## ADR-004: JWT аутентификация с refresh токенами

**Дата:** 2024-02-01  
**Статус:** Принято  
**Участники:** Команда разработки, Security team

### Контекст

Требовалось выбрать систему аутентификации для мессенджера с учетом:
- Безопасности
- Масштабируемости
- Опыта пользователя
- Поддержки офлайн режима

### Рассмотренные варианты

1. **Сессии в БД:**
   - Плюсы: Простота отзыва, контроль
   - Минусы: Нагрузка на БД, сложность масштабирования

2. **JWT только access токены:**
   - Плюсы: Stateless, простота
   - Минусы: Невозможность отзыва, безопасность

3. **JWT с refresh токенами:**
   - Плюсы: Баланс безопасности и производительности
   - Минусы: Сложнее реализации

### Решение

Выбрать **JWT с refresh токенами** по следующим причинам:
1. **Безопасность:** Короткоживущие access токены, возможность отзыва
2. **Масштабируемость:** Stateless серверы
3. **Опыт:** Автоматическое обновление токенов
4. **Офлайн:** Refresh токены хранятся локально

### Реализация

- **Access токены:** 15 минут, в памяти клиента
- **Refresh токены:** 7 дней, в системном keyring
- **Хранение:** Refresh токены хешируются в БД
- **Обновление:** Автоматическое перед истечением

### Последствия

**Положительные:**
- Хорошая безопасность
- Легкое масштабирование
- Плавный пользовательский опыт
- Поддержка офлайн

**Отрицательные:**
- Сложность реализации
- Необходимость хранения refresh токенов
- Риск компрометации refresh токенов

## ADR-005: PostgreSQL с SQLx

**Дата:** 2024-02-05  
**Статус:** Принято  
**Участники:** Команда разработки, DBA

### Контекст

Требовалось выбрать базу данных и ORM с учетом:
- Надежности
- Производительности
- Типобезопасности
- Поддержки транзакций

### Рассмотренные варианты

1. **SQLite:**
   - Плюсы: Простота, встроенная
   - Минусы: Ограниченная масштабируемость, нет конкурентности

2. **PostgreSQL + Diesel:**
   - Плюсы: Зрелый ORM, хорошая документация
   - Минусы: Сложнее асинхронности

3. **PostgreSQL + SQLx:**
   - Плюсы: Async, проверка типов на этапе компиляции
   - Минусы: Менее зрелый

### Решение

Выбрать **PostgreSQL + SQLx** по следующим причинам:
1. **Типобезопасность:** Проверка запросов на этапе компиляции
2. **Async:** Полная поддержка async/await
3. **Миграции:** Встроенная система миграций
4. **Производительность:** Подготовленные запросы, connection pooling

### Последствия

**Положительные:**
- Гарантии корректности запросов
- Высокая производительность
- Современная async архитектура
- Надежные транзакции

**Отрицательные:**
- Сложность обучения SQLx
- Меньше документации
- Требует PostgreSQL

## ADR-006: Системный keyring для хранения токенов

**Дата:** 2024-02-10  
**Статус:** Принято  
**Участники:** Команда разработки, Security team

### Контекст

Требовалось выбрать способ хранения refresh токенов на клиенте с учетом:
- Безопасности
- Кроссплатформенности
- Простоты использования
- Восстановления сессии

### Рассмотренные варианты

1. **LocalStorage:**
   - Плюсы: Простота
   - Минусы: Уязвимо к XSS, небезопасно

2. **Зашифрованный файл:**
   - Плюсы: Контроль над шифрованием
   - Минусы: Сложность, риск потери ключа

3. **Системный keyring:**
   - Плюсы: Максимальная безопасность, нативная поддержка
   - Минусы: Зависимость от ОС

### Решение

Выбрать **системный keyring** по следующим причинам:
1. **Безопасность:** Использует встроенные механизмы безопасности ОС
2. **Кроссплатформенность:** Библиотека keyring поддерживает все ОС
3. **Восстановление:** Токены сохраняются между запусками
4. **Изоляция:** Данные защищены от других приложений

### Реализация по платформам

- **Windows:** Credential Manager
- **macOS:** Keychain
- **Linux:** Secret Service (GNOME Keyring/KWallet)

### Последствия

**Положительные:**
- Максимальная безопасность
- Нативная интеграция с ОС
- Автоматическое управление ключами
- Восстановление сессий

**Отрицательные:**
- Зависимость от библиотеки keyring
- Разное поведение на разных ОС
- Сложность отладки

## ADR-007: OpenAPI для документации API

**Дата:** 2024-02-15  
**Статус:** Принято  
**Участники:** Команда разработки, Technical writers

### Контекст

Требовалось выбрать способ документирования API с учетом:
- Актуальности документации
- Автоматизации
- Поддержки клиентских генераторов
- Читаемости

### Рассмотренные варианты

1. **Ручная документация (Markdown):**
   - Плюсы: Полный контроль, гибкость
   - Минусы: Расхождение с кодом, ручное обновление

2. **Swagger/OpenAPI с аннотациями:**
   - Плюсы: Автоматическая генерация, клиентские SDK
   - Минусы: Загрязнение кода аннотациями

3. **utoipa для Rust:**
   - Плюсы: Типобезопасность, интеграция с Axum
   - Минусы: Специфично для Rust

### Решение

Выбрать **utoipa** для генерации OpenAPI спецификации по причинам:
1. **Типобезопасность:** Использование типов Rust для документации
2. **Интеграция:** Прямая интеграция с Axum маршрутами
3. **Автоматизация:** Автоматическая генерация документации
4. **Клиенты:** Генерация TypeScript типов

### Реализация

- **Документация:** Swagger UI на `/swagger-ui/`
- **Спецификация:** OpenAPI JSON на `/api-docs/openapi.json`
- **Клиенты:** Автоматическая генерация TypeScript типов
- **Валидация:** Встроенная валидация запросов

### Последствия

**Положительные:**
- Всегда актуальная документация
- Автоматические клиентские SDK
- Встроенная валидация
- Интерактивная документация

**Отрицательные:**
- Зависимость от utoipa
- Ограничения системы типов
- Сложность кастомизации

## ADR-008: WebSocket для реального времени

**Дата:** 2024-02-20  
**Статус:** Принято (планируется)  
**Участники:** Команда разработки

### Контекст

Требовалось выбрать технологию для реального времени в мессенджере с учетом:
- Производительности
- Надежности
- Масштабируемости
- Поддержки в браузерах

### Рассмотренные варианты

1. **Long Polling:**
   - Плюсы: Простота, обратная совместимость
   - Минусы: Задержки, нагрузка на сервер

2. **Server-Sent Events (SSE):**
   - Плюсы: Простота, встроенная в HTTP
   - Минусы: Только сервер → клиент

3. **WebSocket:**
   - Плюсы: Двусторонняя, низкая задержка, стандарт
   - Минусы: Сложнее реализации

### Решение

Выбрать **WebSocket** по следующим причинам:
1. **Производительность:** Низкая задержка, постоянное соединение
2. **Двусторонность:** Возможность отправки сообщений в обе стороны
3. **Стандарт:** Поддержка во всех современных браузерах
4. **Экосистема:** Хорошая поддержка в Rust (tokio-tungstenite)

### Планируемая архитектура

- **Сервер:** Axum с WebSocket поддержкой
- **Broadcast:** Tokio broadcast channels
- **Аутентификация:** JWT через query параметры
- **Состояние:** In-memory для активных соединений

### Последствия

**Положительные:**
- Минимальная задержка
- Эффективное использование ресурсов
- Современный стандарт
- Хорошая масштабируемость

**Отрицательные:**
- Сложность реализации
- Управление состоянием соединений
- Балансировка нагрузки

## ADR-009: Rate limiting на сервере

**Дата:** 2024-02-25  
**Статус:** Принято  
**Участники:** Команда разработки, Security team

### Контекст

Требовалось защитить API от злоупотреблений с учетом:
- Защиты от брутфорса
- Предотвращения DoS
- Справедливого использования
- Производительности

### Рассмотренные варианты

1. **Без rate limiting:**
   - Плюсы: Простота
   - Минусы: Уязвимость к атакам

2. **Простые счетчики:**
   - Плюсы: Простота реализации
   - Минусы: Нет fairness, проблемы с распределением

3. **Token bucket алгоритм:**
   - Плюсы: Справедливость, гибкость
   - Минусы: Сложнее реализации

### Решение

Выбрать **token bucket алгоритм** через библиотеку `governor` по причинам:
1. **Справедливость:** Гарантирует fair распределение
2. **Гибкость:** Разные лимиты для разных endpoints
3. **Производительность:** Эффективная реализация
4. **Интеграция:** Хорошая интеграция с Axum

### Конфигурация лимитов

- **Аутентификация:** 5 запросов в минуту
- **Регистрация:** 10 запросов в час
- **Общие endpoints:** 100 запросов в минуту
- **WebSocket:** Отдельные лимиты на соединения

### Последствия

**Положительные:**
- Защита от атак
- Справедливое использование
- Гибкая конфигурация
- Хорошая производительность

**Отрицательные:**
- Сложность конфигурации
- Необходимость мониторинга
- Риск ложных срабатываний
