# Чеклист для Code Review - Chewback

## Общие принципы

### Цели Code Review
- ✅ Улучшение качества кода
- ✅ Обмен знаниями в команде
- ✅ Обнаружение дефектов на ранней стадии
- ✅ Поддержание единых стандартов кодирования

### Правила проведения
- **Уважительное общение:** Критикуйте код, а не автора
- **Конструктивная обратная связь:** Предлагайте решения, а не только указывайте на проблемы
- **Своевременность:** Проводите review в течение 24 часов
- **Объем:** Не более 400 строк за один review

## Чеклист по категориям

### 1. Функциональность

#### Корректность
- [ ] Код выполняет заявленную функциональность
- [ ] Обработаны edge cases и граничные условия
- [ ] Нет регрессий существующего функционала
- [ ] Логика соответствует требованиям

#### Обработка ошибок
- [ ] Все возможные ошибки обработаны
- [ ] Пользователь получает понятные сообщения об ошибках
- [ ] Критические ошибки логируются
- [ ] Есть fallback механизмы при сбоях

#### Безопасность
- [ ] Нет уязвимостей (SQL injection, XSS, CSRF)
- [ ] Чувствительные данные защищены
- [ ] Валидация входных данных
- [ ] Проверка прав доступа

### 2. Качество кода

#### Читаемость
- [ ] Код понятен без дополнительных комментариев
- [ ] Осмысленные имена переменных и функций
- [ ] Сложная логика прокомментирована
- [ ] Соблюдены соглашения по именованию

#### Структура
- [ ] Функции и методы выполняют одну задачу
- [ ] Размер функций не превышает 50 строк
- [ ] Нет дублирования кода (DRY)
- [ ] Логически связанный код сгруппирован

#### Сложность
- [ ] Цикломатическая сложность не превышает 10
- [ ] Нет вложенных циклов глубже 3 уровней
- [ ] Условные операторы не слишком сложные
- [ ] Используются паттерны для упрощения сложной логики

### 3. Тестирование

#### Unit тесты
- [ ] Написаны тесты для новой функциональности
- [ ] Покрыты основные сценарии использования
- [ ] Покрыты edge cases
- [ ] Тесты независимы и изолированы

#### Интеграционные тесты
- [ ] Протестировано взаимодействие компонентов
- [ ] Тесты используют моки для внешних зависимостей
- [ ] Тесты не зависят от порядка выполнения
- [ ] Тесты очищают состояние после выполнения

#### Качество тестов
- [ ] Тесты имеют понятные имена
- [ ] Тесты проверяют одно утверждение
- [ ] Нет тестов, которые всегда проходят
- [ ] Тесты не содержат логики бизнес-уровня

### 4. Производительность

#### Эффективность
- [ ] Нет неоптимальных алгоритмов (O(n²) там где можно O(n log n))
- [ ] Минимизировано количество запросов к БД
- [ ] Используется кэширование где это уместно
- [ ] Нет memory leaks

#### Ресурсы
- [ ] Закрываются файловые дескрипторы и соединения
- [ ] Освобождается память
- [ ] Нет бесконечных циклов
- [ ] Ограничены размеры буферов

### 5. Безопасность

#### Данные
- [ ] Пароли хешируются с солью
- [ ] Токены имеют ограниченное время жизни
- [ ] Чувствительные данные не логируются
- [ ] Используются подготовленные SQL запросы

#### Аутентификация и авторизация
- [ ] Проверка прав доступа на всех уровнях
- [ ] Сессии управляются безопасно
- [ ] Нет hardcoded секретов в коде
- [ ] Используются безопасные алгоритмы шифрования

### 6. Архитектура

#### Принципы проектирования
- [ ] Соблюден принцип единственной ответственности
- [ ] Соблюден принцип открытости/закрытости
- [ ] Нет циклических зависимостей
- [ ] Слои приложения четко разделены

#### Зависимости
- [ ] Минимизировано количество зависимостей
- [ ] Используются стабильные версии библиотек
- [ ] Нет неиспользуемых импортов
- [ ] Зависимости актуальны и поддерживаются

### 7. Документация

#### Код
- [ ] Сложные алгоритмы задокументированы
- [ ] Публичные API имеют документацию
- [ ] Примеры использования для сложных функций
- [ ] Обновлена документация при изменении API

#### Коммиты
- [ ] Сообщения коммитов следуют Conventional Commits
- [ ] Коммиты логически разделены
- [ ] Нет коммитов с временным кодом
- [ ] Связаны с issue/ticket

### 8. Специфичные для технологий

#### Rust (Сервер и Tauri)
- [ ] Нет unsafe кода без необходимости
- [ ] Обработка всех Result и Option
- [ ] Использование borrow checker корректно
- [ ] Нет паники в production коде

#### TypeScript/SolidJS (Клиент)
- [ ] Строгая типизация
- [ ] Нет any типов без необходимости
- [ ] Корректное использование реактивности
- [ ] Нет memory leaks в эффектах

#### SQL (Миграции и запросы)
- [ ] Индексы для часто используемых полей
- [ ] Внешние ключи с каскадным удалением
- [ ] Нет N+1 проблем в запросах
- [ ] Используются транзакции для атомарности

### 9. Интеграция

#### Совместимость
- [ ] Код работает на всех поддерживаемых платформах
- [ ] Нет breaking changes в публичном API
- [ ] Обратная совместимость сохранена
- [ ] Миграции данных обработаны

#### Сборка и деплой
- [ ] Код компилируется без ошибок
- [ ] Тесты проходят в CI/CD
- [ ] Нет hardcoded конфигурации
- [ ] Логирование настроено корректно

## Процесс Review

### Перед началом
1. **Подготовка автора:**
   - [ ] Код готов к review
   - [ ] Тесты написаны и проходят
   - [ ] Документация обновлена
   - [ ] Локально протестировано

2. **Подготовка ревьюера:**
   - [ ] Понимание контекста изменений
   - [ ] Знание связанных issue/ticket
   - [ ] Готовность уделить достаточно времени

### Во время Review
1. **Первый проход:**
   - Общее понимание изменений
   - Проверка соответствия требованиям
   - Оценка архитектурных решений

2. **Детальный анализ:**
   - Проверка по чеклисту
   - Анализ сложных участков кода
   - Проверка тестов

3. **Обратная связь:**
   - Критические проблемы (блокирующие)
   - Важные улучшения (рекомендуемые)
   - Мелкие замечания (опциональные)

### После Review
1. **Для автора:**
   - [ ] Исправлены блокирующие проблемы
   - [ ] Рассмотрены рекомендации
   - [ ] Ответ на все комментарии
   - [ ] Повторное review при необходимости

2. **Для ревьюера:**
   - [ ] Подтверждение исправлений
   - [ ] Одобрение изменений
   - [ ] Обновление статуса в системе

## Критерии одобрения

### Обязательные (блокирующие)
- [ ] Код выполняет заявленную функциональность
- [ ] Нет критических уязвимостей безопасности
- [ ] Все тесты проходят
- [ ] Соблюдены основные стандарты кодирования

### Рекомендуемые
- [ ] Код читаем и понятен
- [ ] Архитектурные решения обоснованы
- [ ] Документация актуальна
- [ ] Производительность адекватна

### Опциональные
- [ ] Код идеально соответствует всем стандартам
- [ ] Максимальное покрытие тестами
- [ ] Идеальная производительность
- [ ] Полная документация

## Эскалация проблем

### Когда эскалировать
1. **Технические разногласия:** Если нет консенсуса по архитектурному решению
2. **Срочные изменения:** Критические security fixes или hotfixes
3. **Большие изменения:** PR больше 1000 строк кода
4. **Многократные review:** Более 3 итераций без прогресса

### Процесс эскалации
1. Обсудить в команде
2. Привлечь технического лида
3. Задокументировать решение
4. Обновить стандарты при необходимости

## Best Practices

### Для авторов
- Разбивайте большие PR на меньшие
- Пишите понятные описания изменений
- Отвечайте на все комментарии
- Учитесь на feedback

### Для ревьюеров
- Будьте конструктивны
- Объясняйте почему, а не только что
- Уважайте время автора
- Учите, а не критикуйте

### Для команды
- Регулярные ретроспективы процесса
- Обновление стандартов по мере необходимости
- Обмен знаниями и лучшими практиками
- Поддержка культуры качественного кода

## Шаблоны комментариев

### Похвала
```
Отличная работа! Особенно понравилось:
- Четкая структура кода
- Хорошие тесты
- Понятные комментарии
```

### Конструктивная критика
```
Предлагаю рассмотреть альтернативу:
**Текущий подход:** [описание]
**Проблема:** [в чем проблема]
**Предложение:** [альтернативное решение]
**Преимущества:** [почему лучше]
```

### Вопросы
```
Не совсем понял этот момент:
- Как работает [конкретная часть]?
- Что происходит при [специфический случай]?
- Почему выбран именно этот подход?
```

### Безопасность
```
Обнаружена потенциальная уязвимость:
**Тип:** [SQLi/XSS/CSRF/etc]
**Место:** [файл:строка]
**Риск:** [описание риска]
**Исправление:** [как исправить]
```

## Метрики качества

### Отслеживаемые метрики
1. **Время до первого review:** < 24 часов
2. **Количество итераций:** в среднем 1-2
3. **Размер PR:** < 400 строк
4. **Покрытие тестами:** > 80%
5. **Количество комментариев:** 5-15 на PR

### Анализ эффективности
- Регулярные опросы удовлетворенности
- Анализ времени цикла разработки
- Отслеживание дефектов в production
- Обучение на основе common issues
